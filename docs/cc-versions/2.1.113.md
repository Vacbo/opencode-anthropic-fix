## Distribution Change (MAJOR)

v2.1.113 introduces a **native-binary distribution** per the upstream changelog:

> "Changed the CLI to spawn a native Claude Code binary (via a per-platform optional dependency) instead of bundled JavaScript"

### npm Package Structure Shift

| Field                  | 2.1.112                           | 2.1.113                                                                                                      |
| ---------------------- | --------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `bin.claude`           | `cli.js` (13.7 MB JS bundle)      | `bin/claude.exe` (500-byte stub)                                                                             |
| Main package unpacked  | 49.3 MB                           | **132 KB** (99.7% reduction)                                                                                 |
| `optionalDependencies` | `@img/sharp-*` (image processing) | `@anthropic-ai/claude-code-{darwin,linux,win32}-{arm64,x64}` (8 platform-specific binaries, 195-246 MB each) |

The real application now lives in **per-platform Bun-compiled binaries**. The main npm package is a thin shim with `install.cjs` that copies the correct platform binary over the `bin/claude.exe` stub.

### Binary Format: Bun-compiled SEA

The native binary is confirmed Bun `--compile` output:

- **Mach-O section**: `__BUN/__bun` (macOS)
- **ELF section**: `.bun` (Linux)
- **PE section**: `.bun` (Windows)
- **Trailer**: `\n---- Bun! ----\n` at binary end
- **Embedded**: Full minified JavaScript source + JavaScriptCore + ripgrep + seccomp helpers

The JavaScript source is stored as NUL-terminated plaintext within the `StandaloneModuleGraph`. Extraction via `strings` or `bun-demincer` recovers the same content that the old `cli.js` contained.

### Our Extraction Pipeline Impact

`scripts/analysis/extract-cc-bundle.ts` was extended to branch on version:

- **<2.1.113**: legacy `cli.js` extraction from npm tarball (unchanged)
- **≥2.1.113**: download platform-specific tarball → extract native binary → `strings -n 10` → feed to existing `extract-fingerprint.ts` regex pipeline

The regex pipeline in `extract-fingerprint.ts` works **unchanged** against the `strings` output because the JS source is stored verbatim in the binary. Verified: 30 beta flags, 2 OAuth client IDs, 5 scopes — all extracted successfully.

See commit message and `.sisyphus/evidence/phase-1-claim-validation/2026-04-17/SANITY-CHECK-FINDINGS.md` for the full investigation.

## Alignment Notes for This Repository

The plugin auto-fetches `dist-tags.latest` from npm at startup (per `src/headers/user-agent.ts`), so the UA version bump to `2.1.113` is already handled without source changes — verified via live debug output on 2026-04-17.

**Plugin-side changes required for this release**:

1. ✅ `scripts/analysis/extract-cc-bundle.ts` — extended to handle both cli.js and Bun-binary distributions (version-branched at `NATIVE_BINARY_MIN_VERSION = "2.1.113"`)
2. ⚠️ `scripts/analysis/extract-fingerprint.ts` — regex pipeline works unchanged against `strings` output, but two fields (UA template, billing salt) extract with `null` because template literals in JS don't materialize as contiguous strings. Workaround: use live-runtime-capture values for these two fields. Proper fix deferred (not blocking).
3. ⚠️ `scripts/drift-checker/check-cch-constants.ts` — npm-mode still downloads cli.js; this mode is NOT USED for 2.1.113+. Standalone-binary mode works as before. Remove the npm-mode fallback or update it to use the new extractor.
4. ✅ The runtime behavior of the plugin is unaffected — we mimic the WIRE format, which has not changed.
